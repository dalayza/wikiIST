Pasos para la atención de incidencias en SIA Producción

A continuación describiremos paso a paso de como se debe atender en una situación de incidencia en el aplicativo SIA ambiente de producción. Para ello, nombraremos paso a paso con un orden de atención para la verificación y análisis de la incidencia reportada:


1.- Servicios Desconectados:

- Lo primero que se debe realizar es verificar la desconexión del canal o de los canales. Para ello, verificamos las conexiónes de cada canal de comunicación en el menú ingresando con el comando cpmenu.

Ahora, si bien se puede observar que alguna conexión presenta el mensaje de error en el campo de observaciones, debemos llamar al personal de GRUPO OPERACIONES para que realice el respectivo sing on al módulo afectado y así volver a reestablecer la comunicación.

Si aún no podemos logramos establecer la comunicación o servicio, verificamos acceder al servicio servidor con el comando telnet "IP" "PORT" y verificar que la comunicación se haya establecido por dicho puerto.

En caso que ninguno de los pasos anteriores no aplique, debemos de comunicarnos con el personal de FIREWALL y PERIMETRO para verificar la traza de comunicación entre ambos servicios.


2.- DBA Caído:

- Se verifica la disponibilidad de la base de datos. Para ello, se debe ejecutar el comando cpsql, si con el comando ejecutado logramos ingresar a la pantalla de código de la base de datos, quiere decir que dicha base de datos se encuentra correcta. De igual forma y para evitar riesgo, podemos consultar una tabla con el query desc tta080 y consultamos los registros con el query select * from tta080 where rownum < 2;.

Otra manera quela aplicación SIA reconozca que la base de datos se encuentra corrupta, es cuando necesitamos subir la aplicación en la opción cpmenu y la misma muestra un mensaje de ORACLE NO DISPONIBLE.

Cualquier falla relacionada con la base de datos, debe ser reportada al personal de DBA_PM.


3.- No hay respuesta del host:

- Si al verificar las operaciones vivas e indican NO HAY RESPUESTA DEL HOST y de igual forma al verificar los logs de cada transacción no logramos visualizar el mensaje de respuesta de MTI 0110, pudiera ser que el canal de comunicación COLAS MQ de request se encuentra caído. Para ellos, se debe ingresar a la línea de comado de las colas QV1P de SIA con el comando runmqsc QV1P y levantamos la el canal caído; primero se verifica la cola caída con el comando dis chs (*), verificamos el canal caído y luego lo levantamos con el comando start channel (QV0PTOQV1P.CH.P) o el comando start channel (QV1PTOQV0P.CH.P) dependiendo del canal afectado.


4.- Error cryptográfico no reintentar:

Si al verificar las operaciones indican el mensaje de respuesta ERROR DE CRYPTOGRAFIA NO REINTENTAR, lo que debemos de hacer es ingresar a la ruta /compila/bin y ejecutar el comando rmcolas.sh, allí debe indicar la cola que ha tenido afectación de transaccionalidad y por el cual debemos de limpiar indicando Yes en la limpieza de las colas HSIM HCRYP HCRYTO.


Todo servicio se debe confirmar con su respectivo porcentaje de aprobación informado por cada una de las franquicias.

Mastercard

90%
